# 顶部显示访问者 IP 与时间方案

## 目标

登录后的页面顶部显示当前访问者公网 IP 和当前时间，让用户打开书签导航页时能快速确认当前网络出口与时间状态。

## 用户可见结果

- 页面顶部栏显示 `IP: <访问者公网 IP>`。
- 页面顶部栏显示当前时间。
- IP 正在加载或获取失败时，不显示 IP 文案、不显示占位符、不影响页面使用。
- 未登录时仍只显示登录页，不显示 IP 和主页面顶部信息。

## 推荐方案

采用 Cloudflare Pages Function 新增 `/api/ip` 接口，从请求头读取访问者 IP，前端在登录成功并进入主页面后请求该接口，再把成功获取到的 IP 显示到顶部栏。

时间显示由前端本地计算并定时刷新，不依赖后端接口。

## 为什么这样做

Cloudflare Pages 在线上请求中会提供访问者 IP 相关请求头，后端读取比前端直接调用第三方 IP 服务更稳定，也避免引入外部服务依赖。

时间属于浏览器本地展示信息，直接在前端计算即可，不需要增加后端复杂度。

## 对用户的影响

- 用户能在页面顶部看到当前网络出口公网 IP。
- 用户能看到当前时间。
- IP 获取失败不会出现错误提示或空占位，页面保持干净。
- 书签数据、登录逻辑、KV 存储不受影响。

## 替代方案

| 方案 | 优点 | 缺点 | 结论 |
| --- | --- | --- | --- |
| 前端直接请求 `api.ipify.org` | 改动最少，本地也容易看到公网 IP | 依赖第三方服务，失败会受外部网络影响 | 不采用 |
| 后端 `/api/ip` 读取 Cloudflare 请求头 | 不依赖第三方，线上来源更稳定 | 本地开发环境不一定能显示真实公网 IP | 采用 |
| 只在前端尝试读取环境信息 | 几乎不改后端 | 浏览器无法可靠获得访问者公网 IP | 不采用 |

## 实现范围

1. 新增 `functions/api/ip.ts`。
   - 读取 `CF-Connecting-IP`。
   - 本地或代理环境下兼容 `x-forwarded-for`。
   - 成功返回 `{ "ip": "x.x.x.x" }`。
   - 无法取得 IP 时返回非 200 或空错误结果，前端不展示 IP。

2. 修改 `web/src/App.tsx`。
   - 登录并加载主页面后请求 `/api/ip`。
   - 只有成功拿到非空 IP 时才在顶部栏显示。
   - 请求中和失败时都不显示 IP 区域。
   - 在顶部栏显示当前时间，并每秒刷新。

3. 如顶部栏空间不足，轻微调整布局。
   - 保持现有纸质黑白风格。
   - 不重做整体 UI。
   - 不影响退出按钮和标题识别。

## 不做什么

- 不引入第三方 IP 查询服务。
- 不新增数据库或 KV 字段。
- 不保存用户 IP。
- 不在登录页显示 IP。
- 不在 IP 加载中或失败时显示占位文本。

## 验收标准

1. 登录后，顶部栏显示当前时间。
2. `/api/ip` 成功返回 IP 时，顶部栏显示 `IP: <访问者公网 IP>`。
3. `/api/ip` 加载中时，顶部栏不显示 IP 相关文案。
4. `/api/ip` 失败时，顶部栏不显示 IP 相关文案，页面功能正常。
5. 未登录时仍显示登录页，不出现 IP 和主页面顶部信息。
6. `npm run build` 通过。

## 测试路径

- 正常路径：登录后请求 `/api/ip` 成功，顶部同时显示 IP 和时间。
- 加载路径：接口请求未完成时，不显示 IP。
- 错误路径：接口返回错误或网络失败时，不显示 IP，书签功能不受影响。
- 权限路径：未登录访问主页面时，仍由现有鉴权流程进入登录页。
- 时间路径：页面停留超过一秒后，时间会刷新。

## 风险与处理

本地开发环境可能没有 `CF-Connecting-IP`，看到的 IP 可能是代理 IP、本机 IP，或不显示。这个风险只影响本地验证，不影响 Cloudflare Pages 线上效果。线上验收以部署后的访问结果为准。

## 回滚方案

删除 `functions/api/ip.ts`，移除 `web/src/App.tsx` 中的 IP 请求和顶部展示逻辑即可。该功能不写入 KV，不涉及数据迁移。
